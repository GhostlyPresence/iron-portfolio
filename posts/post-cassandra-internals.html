What Happens When Cassandra Receives a Write Request

A beginner-friendly walkthrough of how Cassandra stores data — Journey through commit log to memtables, and SSTables

Distributed systems can feel mysterious at first. Data goes in somewhere, and somehow it ends up safely stored across machines. Cassandra is one of those systems that quietly does a lot of work behind the scenes to make writes fast, durable, and scalable.

In this post we’ll walk through what actually happens when Cassandra receives a write request. Step by step. From the moment a request arrives at a node, to how it is stored in memory, written to disk, and eventually cleaned up through compaction.

Think of it as following a single piece of data through Cassandra’s storage engine.

The Beginning: A Client Sends a Write

Imagine a simple table storing chat messages.

CREATE TABLE messages (
    chat_id text,
    message_time timestamp,
    message text,
    PRIMARY KEY (chat_id, message_time)
);

A client now sends a write request.

INSERT INTO messages (chat_id, message_time, message)
VALUES ('chat123', '2026-01-01 10:00:00', 'Hello Cassandra!');

The request first arrives at one Cassandra node. This node becomes the coordinator for the request.

The coordinator’s job is simple:

  • Figure out which nodes should store the data
  • Send the write to those replica nodes
  • Wait for enough responses based on the consistency level

Once the coordinator knows the replica nodes, the real storage work begins.

Step 1: Writing to the Commit Log

The first place Cassandra writes your data is the Commit Log.

Think of the commit log like a journal. Every change is written there first.

Memory can disappear if a machine crashes. The commit log ensures that Cassandra can always recover recent writes.

The write flow begins like this:

Write Request
      ↓
Append to Commit Log

Important properties of the commit log:

  • It is append-only
  • Writes are sequential on disk
  • Sequential disk writes are extremely fast

Cassandra avoids random disk writes and simply adds new entries to the end of the file.

What “Append-Only” Means in the Commit Log

When Cassandra writes to the commit log, it never modifies existing data inside the file. Instead, it simply adds new entries to the end of the file.

In other words:
old data stays untouched, new data is appended at the end

Think of it like writing in a notebook. You don’t go back and erase previous pages when you add a new note. You just write the new note on the next empty line.

Let see this in action with an example:

First Write: 
INSERT user1 → "Manas"
Commit log file: 
[user1, Manas]
Second Write: 
INSERT user2 → "Aparna"
Commit log file: 
[user1, Manas], 
[user2, Aparna]
Third Write: 
UPDATE user1 → "Manas Sinha"
Commit log file: 
[user1, Manas]
[user2, Aparna]
[user1, Manas Sinha]
Cassandra still does not modify the old entry. It simply appends a new one.

Why Cassandra Uses Append-Only Logs

There are 2 main reasons why Cassandra uses an append-only design for the commit log:

1. Disk Performance

Modern hard drives and SSDs are optimized for sequential writes. When data is written in a continuous stream, the disk can write it much faster than if it had to jump around to modify existing data.

2. Crash recovery

The commit log ensures durability. If a node crashes, Cassandra can replay the commit log to recover any recent writes that were not yet flushed to disk.

This design allows Cassandra to provide strong durability guarantees while still keeping writes fast.

Step 2: Writing to the MemTable

After the commit log write succeeds, Cassandra stores the data in the MemTable.

A MemTable is an in-memory data structure that temporarily stores recent writes.

Internally it behaves like a sorted map. Rows are organized like this: 

Partition Key → Clustering Key → Row Data

If messages arrive in random order, Cassandra still keeps them sorted inside the MemTable. 

chat1
  ├─ 10:00 → "Hi"
  ├─ 10:01 → "Hello"
  └─ 10:02 → "What's up?"

chat2
  ├─ 10:03 → "How are you?"
  └─ 10:05 → "Hey!"

Two key properties:

  • Rows are grouped by partition key
  • Rows inside a partition are sorted by clustering key

Step 3: Flushing the MemTable

MemTables cannot grow forever. Eventually Cassandra must write them to disk.

This happens when:

  • The MemTable reaches its size limit
  • Memory pressure triggers a flush
  • The system decides to flush periodically

When flushing happens, Cassandra writes the MemTable to disk as an SSTable.

Step 4: SSTables

SSTable stands for Sorted String Table. It is the main on-disk storage format Cassandra uses.

An SSTable is:

  • Immutable
  • Sorted
  • Stored on disk

Because the MemTable was already sorted, Cassandra simply writes the data sequentially to disk. 

SSTable

Partition: chat1
  10:00 → Hi
  10:01 → Hello
  10:02 → What's up?

Partition: chat2
  10:03 → How are you?
  10:05 → Hey

Once the flush completes, Cassandra can clear the MemTable and reuse that memory.

Updates and Deletes

Cassandra does not update rows in place. Instead, every change is simply written again with a newer timestamp.

This means:

  • INSERT is a write
  • UPDATE is a write
  • DELETE is also a write

When Cassandra reads data later, it merges rows from multiple SSTables and returns the newest version.

Tombstones

Deletes in Cassandra do not immediately remove data. Instead the system writes a marker called a tombstone. 

(chat123, 10:00) → TOMBSTONE

This marker ensures that deleted data will eventually be removed across all replicas in the cluster.

Compaction

Over time Cassandra accumulates many SSTables. Some contain old values, others contain newer ones.

To keep the system efficient, Cassandra runs a background process called compaction.

During compaction Cassandra:

  • Reads multiple SSTables
  • Merges their contents
  • Applies tombstones
  • Writes a new optimized SSTable

After compaction the old SSTables are deleted.

A Concrete Example of Compaction

To really understand compaction, it helps to look at a small example. Imagine Cassandra has already flushed several MemTables to disk, creating three SSTables.

These SSTables might contain different versions of the same row because Cassandra never updates files in place.

Suppose our table contains chat messages like this:

PRIMARY KEY (chat_id, message_time)

Over time Cassandra might produce SSTables like the following.

SSTable 1

Partition: chat1
  10:00 → "Rest in peace my granny"
  10:01 → "She got hit by a bazooka"

Partition: chat2
  10:03 → "Kabloom"

Later, someone edits a message and deletes another. Cassandra writes new entries instead of modifying the existing SSTable.

SSTable 2

Partition: chat1
  10:01 → "Kabloom Kablow"   (updated message)

SSTable 3

Partition: chat1
  10:00 → TOMBSTONE       (message deleted)

At this point Cassandra now has three SSTables that contain overlapping pieces of data.

If a read happens for chat1, Cassandra must merge data from multiple SSTables to determine the correct result.

This is where compaction helps.

During Compaction

Cassandra reads rows from all SSTables and resolves them using timestamps.

  • The newer message replaces the older version.
  • The tombstone hides the deleted row.

The merge looks like this internally:


Input SSTables
--------------

SSTable1
chat1
 10:00 → "Rest in peace my granny"
 10:01 → "She got hit by a bazooka"

SSTable2
chat1
 10:01 → "Kabloom Kablow"

SSTable3
chat1
 10:00 → TOMBSTONE

Cassandra processes the rows in sorted order and applies the newest timestamps.

Resulting SSTable After Compaction

SSTable_new

Partition: chat1
  10:01 → "She got hit by a bazooka"

Partition: chat2
  10:03 → "Kabloom Kablow"

The row (chat1, 10:00) disappeared because the tombstone indicated it was deleted.

Once the compaction finishes, Cassandra deletes the old SSTables and keeps the new compacted one.

The result is fewer files, cleaner data, and faster reads for future queries.

The Entire Write Path

Client Write
      ↓
Coordinator Node
      ↓
Commit Log (durability)
      ↓
MemTable (in memory)
      ↓
Flush
      ↓
SSTable (on disk)
      ↓
More writes → more SSTables
      ↓
Compaction merges them

Every step is designed with one goal: make writes extremely fast while maintaining durability and consistency.

Final Thoughts

Cassandra’s storage engine may feel unusual if you come from traditional databases.

But the philosophy behind it is surprisingly simple:

  • Append new data
  • Never rewrite existing files
  • Merge things later in the background

This design allows Cassandra to handle massive distributed workloads while still keeping writes fast and reliable.

Kafka Explained From First Principles

A beginner-friendly guide to message queues, Kafka architecture, and how modern systems move massive amounts of data reliably.

Part 1 — Beginner: Understanding the Basics

The Problem: Systems Need to Talk

Modern software systems rarely live alone. A typical application might have:
  • API servers
  • payment services
  • notification systems
  • analytics pipelines
  • databases
Every one of these systems needs to exchange information. Example: 

User places order
    ↓
Payment service processes payment
    ↓
Inventory service updates stock
    ↓
Notification service sends email
    ↓
Analytics pipeline records event
If each system called the other directly using APIs, the architecture becomes fragile. If one system is slow or down, everything else breaks.

This is where message queues enter the story.

What Is a Message Queue?

A message queue is simply a system where:
  • one application sends messages
  • another application processes them later
Instead of direct communication: 

Service A → API → Service B
We use: 

Service A → Queue → Service B
The queue stores the message safely until the consumer processes it. Think of it like a mailbox. You drop a letter in the mailbox. The recipient checks it when they are ready. Much less shouting across the street.

Popular Message Queues Used in Industry

Different companies use different systems depending on scale.
  • RabbitMQ — classic message broker
  • Apache Kafka — distributed streaming platform
  • Amazon SQS — managed cloud queue
  • Google Pub/Sub — cloud event streaming
  • NATS — lightweight messaging system

Kafka became popular because it handles extremely large data streams efficiently.

Queue vs API Communication

Understanding this difference is important. API communication is synchronous . 


Service A → API request → Service B
(wait)
Service B responds

Queue communication is asynchronous.


Service A → message queue
Service B → processes message later
Benefits:
  • services are loosely coupled
  • systems can scale independently
  • temporary failures do not break the pipeline
In simple terms: APIs demand immediate attention. Queues politely wait their turn.

What Is Kafka?

Kafka is a distributed event streaming platform.

That sounds complicated, but the idea is simple. Kafka is designed to:
  • accept messages from producers
  • store them reliably
  • deliver them to consumers

Kafka became popular because it handles extremely large data streams efficiently.

Companies using Kafka:
  • LinkedIn
  • Netflix
  • Uber
  • Spotify
  • Airbnb
Kafka pipelines process millions of events per second. Examples of events:
  • user clicks
  • orders
  • payments
  • logs
  • sensor readings

The Kafka Cluster

Kafka runs on multiple machines. This group is called a Kafka cluster. Each server inside the cluster is called a broker. 


Kafka Cluster

Broker 1
Broker 2
Broker 3
Reasons for using multiple brokers:
  • higher throughput
  • fault tolerance
  • horizontal scalability
If one broker crashes, others continue serving data. Kafka is built for systems where downtime is unacceptable.

Topics: Categories of Data

Messages in Kafka are stored inside topics.A topic is simply a category of events.

Example topics:
  • orders
  • payments
  • user-signups
  • click-events
  • logs
Producers write to topics. Consumers read from topics.

Partitions: The Key to Kafka's Scalability

Each topic is divided into partitions. Partitions allow multiple consumers to process messages in parallel.

Example: 

Topic: orders

Partition 0
Partition 1
Partition 2
Partition 3
Messages are distributed across partitions. Benefits:
  • parallel processing
  • horizontal scalability
  • high throughput
Important rule:

Ordering is guaranteed only within a partition.

Producers

Producers send messages into Kafka. Example producer code conceptually: 

producer.send(topic="orders",
              key="order123",
              value="itemA, quantity=2")

The key helps determine which partition the message goes to.

Kafka uses: 

partition = hash(key) % number_of_partitions
This ensures messages with the same key land in the same partition.

Consumers

Consumers read messages from Kafka. Consumers poll Kafka instead of receiving pushed messages. 


consumer.poll()
Why polling? Because consumers control the speed of processing. Kafka never overwhelms slow consumers. Very polite behaviour.

Consumer Groups

Consumers often work in teams.

A consumer group is a set of consumers sharing the work.

Say there is a topic about orders placed on a ecommere website. There are two teams that need this data. Team 1 needs it for analytics and Team 2 needs it for fraud detection. Each team forms a consumer group. This ensures that each group can read the same data independently without interfering with each other.

Example 1: Topic has 6 partitions. Consumer group A has 3 consumers. 

Consumer A → P0, P1
Consumer B → P2, P3
Consumer C → P4, P5
Example 2: Topic has 6 partitions. Consumer group B has 7 consumers. 

Consumer A → P0
Consumer B → P1
Consumer C → P2
Consumer D → P3
Consumer E → P4
Consumer F → P5
Consumer G → Standby (no partition assigned)
Rules:
  • each partition assigned to only one consumer per group.
  • multiple groups can read same topic independently
This allows multiple applications to process the same events. Example:
  • analytics-service
  • fraud-detection-service
  • warehouse-service
All reading the same order events.

Part 2 — Advanced: Kafka Internals

Offsets

Each message in a partition has an offset.

Offsets are sequential numbers. 

Partition 0

Offset 0 → message
Offset 1 → message
Offset 2 → message
Consumers store offsets to know where they left off. Kafka stores offsets inside a special topic: 

__consumer_offsets

Consumer offsets are stored at the consumer group level. Each consumer group maintains its own offset for every partition of a topic. Offsets are stored in Kafka’s internal__consumer_offsets topic.

This allows multiple consumer groups to read the same topic independently without interfering with each other. The total number of stored offsets equals the number of partitions multiplied by the number of consumer groups.

Example:
Topic: orders
Partitions: 4

Consumer groups: 

analytics-service
fraud-service
warehouse-service
Kafka stores offsets like this: 
analytics-service
  partition 0 → offset 50
  partition 1 → offset 60
  partition 2 → offset 40
  partition 3 → offset 10

fraud-service
  partition 0 → offset 55
  partition 1 → offset 62
  partition 2 → offset 42
  partition 3 → offset 11

warehouse-service
  partition 0 → offset 51
  partition 1 → offset 61
  partition 2 → offset 43
  partition 3 → offset 12

In a nutshell, Different readers can read the same book but be on different pages.

Replication and Fault Tolerance

Kafka protects data using replication. Each partition has multiple replicas.

Example: 

Partition P0

Leader → Broker 1
Replica → Broker 2
Replica → Broker 3

All writes go to the leader. Followers replicate the data. If the leader crashes: Another replica becomes leader automatically. This is called leader election.

Delivery Guarantees

Kafka supports three delivery modes. At-most-once At-least-once Exactly-once Summary:
  • at-most-once → possible message loss
  • at-least-once → no loss but duplicates possible
  • exactly-once → no duplicates, no loss
Exactly-once requires:
  • idempotent producers
  • transactions
Used in financial systems where duplicates are unacceptable.

Producer Optimizations

Sending one message at a time over the network is slow. Every message would require: opening a network request, sending headers, sending the message, waiting for acknowledgment.
Kafka solves this using batching. Instead of sending messages individually, the producer collects multiple messages in memory and sends them together in a single request.

Now the flow becomes: 

Producer Buffer

msg1
msg2
msg3
msg4
msg5
   ↓
BATCH
   ↓
single network request


Producer
   ↓
Batch [msg1 msg2 msg3 msg4 msg5]
   ↓
Kafka Broker

Instead of 5 requests, we send 1 request. Much faster.

>> Where Does Batching Happen?

Batching happens inside the producer application, in memory. The producer maintains buffers for each partition.

Example: 

Producer Memory

Partition 0 buffer
msg1
msg3
msg7

Partition 1 buffer
msg2
msg4
msg6

Each partition gets its own batch. When a batch is ready, Kafka sends it to the leader broker of that partition.

>> Two Configurations Control Batching

Two producer settings determine when batches are sent. batch.size Maximum size of a batch in bytes. linger.ms How long the producer waits for more messages before sending the batch.

If batch.size is 1MB and linger.ms is 100ms, the producer will send a batch when either:
  • the batch reaches the maximum size (batch.size)
  • the linger time expires (linger.ms)

Consumer Polling and Fetching

Consumers fetch batches of messages. Important settings:
  • max.poll.records
  • fetch.min.bytes
  • fetch.max.bytes

Consumers also send heartbeats to Kafka to show they are alive.

If a consumer stops heartbeating, Kafka triggers rebalance.

Rebalacing means redistributing the partitions among the available consumers.

Kafka Streams

Kafka Streams allows real-time processing. Example operations:
  • filter events
  • aggregate counts
  • join streams
  • windowed computations
Example pipeline: 

orders topic → stream processor → analytics topic

Streams can maintain state locally using RocksDB. State changes are backed up using changelog topics.

Kafka Connect

Kafka Connect moves data between Kafka and other systems. Example pipelines: 

MySQL → Kafka
Kafka → Elasticsearch
Kafka → Data Warehouse
Connectors eliminate the need to write custom ingestion code. Engineers everywhere are grateful.

Why Kafka Is So Fast

Kafka performance comes from several design choices:
  • append-only log storage
  • sequential disk writes
  • zero-copy network transfer
  • partition parallelism
  • batching and compression

These allow Kafka clusters to handle millions of events per second.

Final Thoughts

Kafka started as a messaging system. Today it powers:
  • event-driven architectures
  • real-time analytics
  • stream processing pipelines
  • microservice communication

At its core Kafka solves one simple problem: moving large amounts of data safely between systems.

Once you understand topics, partitions, producers, and consumer groups, the rest of Kafka becomes surprisingly logical. And that is the moment when distributed systems stop looking terrifying and start looking fascinating. Or at least fascinating enough to keep reading.

Writing Elasticsearch Queries (Step by Step)

A beginner-friendly guide to constructing Elasticsearch Query DSL JSON from scratch.

One of the first things engineers notice when using Elasticsearch is that queries are written in JSON, not SQL.

If you come from a relational database background, this can feel strange at first. Instead of writing something like: 


        SELECT * FROM products WHERE brand = 'Nike'

Elasticsearch requires a JSON structure describing the query. At first glance this JSON can look intimidating, but the structure is actually very logical once you build it step by step. In this article we will start with a simple document and gradually build a realistic Elasticsearch query.

The Example Document

Before writing queries, we need to understand what our data looks like.

Imagine we have an Elasticsearch index called products containing documents like this: 


        {
          "title": "Running Shoes",           --- This field is of data type 'text'
          "brand": "Nike",                    --- This field is of data type 'keyword'
          "price": 1999,                      --- This field is of data type 'integer'
          "city": "Bengaluru",                --- This field is of data type 'keyword'
          "in_stock": true                    --- This field is of data type 'boolean'
        }

Each document represents a product. In real systems there might be millions of such documents.

Our goal is to search through these documents and find the ones that match certain conditions.

The Basic Structure of an Elasticsearch Query

Every Elasticsearch search request contains a top-level query field. 


{
  "query": {}
}

Think of this field as the place where you describe the conditions documents must satisfy.

Everything about your search will live inside this object.

Step 1 — Searching Text with a Match Query

Suppose we want to find products whose title contains the words running shoes.

In Elasticsearch, text searches are usually done using a match query. 


          {
            "query": {
              "match": {
                "title": "running shoes"
              }
            }
          }

The match query analyzes the text and finds documents containing similar words.
One thing to note is that match query can only be applied on fields that has the data type of 'text'.
Refer to the previous article for more details on data types in Elasticsearch.

So, the query above could match documents like:

  • "title": "Running Shoes"
  • "title": "Lightweight Running Shoes"
  • "title": "Shoes for Running"

This works because Elasticsearch breaks text into searchable tokens. So while searching for running shoes, it will break this into two tokens: "running" and "shoes", and find documents that contain either of these tokens in the title.
The match query does not require an exact match of the whole phrase. It will match documents that contain either "running" or "shoes" in the title.


As you remember from the previous article, Elasticsearch uses an inverted index to quickly find the documents that contain these tokens.

Say the inverted index looks like this: 


            "running" → [doc1, doc2]
            "shoes" → [doc1, doc2, doc3]
Then the match query will find documents 1, 2, and 3 because they contain either "running" or "shoes".

Step 2 — Combining Conditions with a Bool Query

In real applications we rarely search using only one condition.

We usually want to combine multiple conditions together.

Elasticsearch provides a special query called the bool query for this purpose.

A bool query allows us to combine conditions using logic similar to AND / OR in SQL. 


          {
            "query": {
              "bool": {
                "must": [
                  { "match": { "title": "running shoes" } }
                ]
              }
            }
          }

The must clause means the condition must match for a document to be included in the results.

Step 3 — Adding Exact Filters

Now suppose we only want products from the brand Nike.

For exact matches, Elasticsearch uses a term query. Similar to match query, term query can only be applied on fields that has the data type of 'keyword'.

So if the brand field is of type 'keyword', we can write a term query like this: 


{
  "term": {
    "brand": "Nike"
  }
}

Instead of placing this inside must, we typically place exact conditions inside a filter clause. 


{
  "query": {
    "bool": {
      "must": [
        { "match": { "title": "running shoes" } }
      ],
      "filter": [
        { "term": { "brand": "Nike" } }
      ]
    }
  }
}

Filters are faster because they do not compute relevance scores.

What Is a Relevance Score I hear you ask :)

When Elasticsearch returns search results, it does not simply return documents that match the query. It also tries to determine how well each document matches the query.

This is done using something called a relevance score. Every document returned by a search query receives a numeric score that represents how relevant that document is to the search terms.

Documents with higher scores appear earlier in the results. 


        {
          "hits": [
            {
              "_score": 3.42,
              "_source": {
                "title": "Lightweight Running Shoes"
              }
            },
            {
              "_score": 2.87,
              "_source": {
                "title": "Running Shoes for Beginners"
              }
            }
          ]
        }

The _score field is the relevance score. Elasticsearch sorts results by this value so that the most relevant documents appear first.

Why Relevance Scoring Is Needed

Imagine we search for the phrase "running shoes". Suppose our index contains the following product titles:

  • Running Shoes for Marathon Training
  • Comfortable Shoes for Walking
  • Best Lightweight Running Shoes

All three documents contain at least one of the words from the query. However, some clearly match better than others.

Elasticsearch assigns scores to each document to rank them by relevance. 


        Best Lightweight Running Shoes      score = 4.1
        Running Shoes for Marathon Training score = 3.6
        Comfortable Shoes for Walking       score = 0.9

The documents that match the query terms more closely receive higher scores and appear earlier in the results.

What Influences the Score?

Elasticsearch uses a ranking algorithm called BM25 to compute relevance scores. You do not need to know the full formula, but a few key ideas help explain how ranking works.

1. Term Frequency

Documents where the search terms appear more frequently tend to receive higher scores.

2. Field Length

Matches in shorter fields often receive slightly higher scores because the terms are more concentrated.

That means that even if "running shoes" was there in both titles: "Best Lightweight Running Shoes" and "Running Shoes for Marathon Training", the first one would get a higher score because the title is shorter and the terms are more concentrated.

If the terms that you are searching for make up a larger portion of the text, Elasticsearch considers it a stronger match.

3. Rare Words Matter More

Rare terms are more informative than very common words. For example, if a user searches for "carbon plate running shoes", the word carbon might strongly influence the ranking because it appears in fewer documents.

4. Multiple Matching Terms

Documents containing more of the query terms generally receive higher scores than those matching only a subset of the words.

When Scores Are Not Used

Not every part of an Elasticsearch query affects the relevance score. Conditions placed inside the filter clause only determine whether a document matches or not.

For example: 


        {
          "term": {
            "brand": "Nike"
          }
        }

This filter simply checks whether the brand is Nike. It does not influence how documents are ranked.

This separation between filtering and scoring allows Elasticsearch to run searches efficiently while still producing well-ranked results.

With that in mind...

Step 4 — Filtering by Price

Suppose we want products whose price is between 1500 and 3000.

Elasticsearch provides a range query for numeric fields. 


{
  "range": {
    "price": {
      "gte": 1500,
      "lte": 3000
    }
  }
}

Here:

  • gte means "greater than or equal"
  • lte means "less than or equal"

We can add this to our filter list: 


{
  "query": {
    "bool": {
      "must": [
        { "match": { "title": "running shoes" } }
      ],
      "filter": [
        { "term": { "brand": "Nike" } },
        { "range": { "price": { "gte": 1500, "lte": 3000 } } }
      ]
    }
  }
}

Step 5 — Multiple Possible Values

Finally, imagine we want products available in either Bengaluru or Delhi.

For this we can use a terms query. 


{
  "terms": {
    "city": ["Bengaluru", "New Delhi"]
  }
}

This query matches documents where the city field contains any of the listed values.

The Final Query

Putting everything together, we get the following query: 


{
  "query": {
    "bool": {
      "must": [
        { "match": { "title": "running shoes" } }
      ],
      "filter": [
        { "term": { "brand": "Nike" } },
        { "range": { "price": { "gte": 1500, "lte": 3000 } } },
        { "terms": { "city": ["Bengaluru", "New Delhi"] } }
      ]
    }
  }
}

In plain English, this query means:

Find products related to "running shoes", where the brand is Nike, the price is between 1500 and 3000, and the city is either Bengaluru or New Delhi.

A Simple Way to Think About Elasticsearch Queries

The easiest way to understand Elasticsearch queries is to think of them as a process of gradually narrowing down documents.

First we search broadly using text queries, and then we apply filters that progressively reduce the candidate documents.

Once you understand this pattern, writing queries in Elasticsearch becomes much easier.

Understanding Elasticsearch — Part 2 (Internals)

How queries are executed, why doc values exist, and what actually happens inside the cluster when you search.

In the first part of this series we covered the basics of Elasticsearch — documents, indices, inverted indices, and how search works at a conceptual level. And then we took a little detour on how elastic search query works.

In this article we will go deeper into the mechanics of the system. Understanding these internals helps explain why Elasticsearch is fast, how it scales, and what trade-offs it makes compared to traditional databases.

The main topics we will explore are:

  • Query planning and execution
  • Doc values and column-oriented storage
  • Segments, refresh, and merging
  • Shard-level distributed search
  • How Elasticsearch separates filtering and scoring

Query Planning in Elasticsearch

Unlike relational databases, Elasticsearch does not use a traditional cost-based SQL query planner. Instead, it performs a series of query rewriting and optimization steps before executing the search.

When a request reaches Elasticsearch, it goes through several stages:

  1. Query parsing
  2. Query rewriting
  3. Shard routing
  4. Query phase
  5. Fetch phase

The goal of the query planner is to minimize the amount of work required to identify matching documents.

Query Rewriting

High-level queries written using Elasticsearch’s Query DSL are internally rewritten into lower-level Lucene queries. For example, a simple match query: 

          {
            "match": {
              "title": "running shoes"
            }
          }

might be rewritten internally into something closer to a boolean query containing term queries for each token: 

          title:running OR title:shoes

This transformation allows Elasticsearch to operate directly on the inverted index. Query rewriting also enables other optimizations such as collapsing filters and eliminating unnecessary scoring steps.

Filter Context vs Query Context

One of the most important optimizations in Elasticsearch is the distinction between query context and filter context.

Query context computes relevance scores. Filter context only checks whether a document matches a condition.

For example: 

        {
          "bool": {
            "must": [
              { "match": { "title": "shoes" } }
            ],
            "filter": [
              { "term": { "brand": "Nike" } }
            ]
          }
        }

In this case:

  • The match query contributes to scoring.
  • The term filter simply narrows the candidate set.

Filters are faster because they avoid score computation and can be cached.

Doc Values: The Other Side of the Index

The inverted index is optimized for answering questions like:
>“Which documents contain this term?”

But many queries require the opposite:
>“What is the value of this field for each document?”

This is where doc values come in.

Doc values are a column-oriented storage structure that stores field values by document rather than by term.
Conceptually, doc values look like this: 

          DocID | price
          -------------
          1     | 1200
          2     | 900
          3     | 3000
This structure enables efficient operations such as:
  • sorting
  • aggregations
  • scripting
  • grouping

Without doc values, Elasticsearch would have to reconstruct these values from the inverted index, which would be extremely expensive.

Segments and Near Real-Time Search

Elasticsearch achieves near real-time search using a structure called segments.
Each shard is composed of multiple immutable segments. When documents are indexed, they are first written to in-memory buffers and the transaction log.

During a refresh operation, these buffers are converted into a new segment that becomes searchable.
This is why Elasticsearch is described as near real-time rather than strictly real-time.

Over time many small segments accumulate, so Elasticsearch periodically merges them into larger ones. This process improves search performance and removes deleted documents.

Distributed Search: The Two-Phase Execution

Elasticsearch is designed to run across multiple machines, so a single query must often be executed across many shards.
Search execution happens in two main phases.

1. Query Phase

The coordinating node sends the query to all relevant shards. Each shard computes the top matching documents and returns their document IDs and scores.

2. Fetch Phase

Once the top results are known, the coordinating node asks the relevant shards to return the full document contents.
This two-phase design minimizes network traffic and ensures that only the most relevant documents are retrieved.

Shards and Replicas

To scale horizontally, Elasticsearch divides each index into primary shards.
Each shard contains a subset of the documents in the index. Queries can therefore run in parallel across shards.

Elasticsearch also supports replica shards. These replicas serve two purposes:

  • Fault tolerance
  • Increased search throughput

Because replicas can handle search requests, Elasticsearch can distribute query load across multiple nodes.

Putting It All Together

When you send a search request to Elasticsearch, the system performs a sequence of carefully optimized steps:

  1. The query is parsed and rewritten.
  2. Filters narrow the candidate documents.
  3. The inverted index identifies matching documents.
  4. Doc values are used for sorting and aggregations.
  5. Each shard returns its top matches.
  6. The coordinating node merges and returns the final results.

This architecture allows Elasticsearch to search massive datasets in milliseconds while remaining horizontally scalable.

Closing Thoughts

Elasticsearch sits at an interesting intersection of search engines, distributed systems, and analytics platforms.

Understanding concepts like inverted indices, doc values, segments, and distributed query execution makes it much easier to reason about performance and system design decisions.

While many applications treat Elasticsearch as a black box, the engineers who understand its internals are able to design far more efficient search systems.

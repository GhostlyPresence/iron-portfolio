Part 2 — Advanced: Kafka Internals

Offsets

Each message in a partition has an offset.

Offsets are sequential numbers. 

Partition 0

Offset 0 → message
Offset 1 → message
Offset 2 → message
Consumers store offsets to know where they left off. Kafka stores offsets inside a special topic: 

__consumer_offsets

Consumer offsets are stored at the consumer group level. Each consumer group maintains its own offset for every partition of a topic. Offsets are stored in Kafka’s internal__consumer_offsets topic.

This allows multiple consumer groups to read the same topic independently without interfering with each other. The total number of stored offsets equals the number of partitions multiplied by the number of consumer groups.

Example:
Topic: orders
Partitions: 4

Consumer groups: 

analytics-service
fraud-service
warehouse-service
Kafka stores offsets like this: 
analytics-service
  partition 0 → offset 50
  partition 1 → offset 60
  partition 2 → offset 40
  partition 3 → offset 10

fraud-service
  partition 0 → offset 55
  partition 1 → offset 62
  partition 2 → offset 42
  partition 3 → offset 11

warehouse-service
  partition 0 → offset 51
  partition 1 → offset 61
  partition 2 → offset 43
  partition 3 → offset 12

In a nutshell, Different readers can read the same book but be on different pages.

Replication and Fault Tolerance

Kafka protects data using replication. Each partition has multiple replicas.

Example: 

Partition P0

Leader → Broker 1
Replica → Broker 2
Replica → Broker 3

All writes go to the leader. Followers replicate the data. If the leader crashes: Another replica becomes leader automatically. This is called leader election.

Delivery Guarantees

Kafka supports three delivery modes. At-most-once At-least-once Exactly-once Summary:
  • at-most-once → possible message loss
  • at-least-once → no loss but duplicates possible
  • exactly-once → no duplicates, no loss
Exactly-once requires:
  • idempotent producers
  • transactions
Used in financial systems where duplicates are unacceptable.

Producer Optimizations

Sending one message at a time over the network is slow. Every message would require: opening a network request, sending headers, sending the message, waiting for acknowledgment.
Kafka solves this using batching. Instead of sending messages individually, the producer collects multiple messages in memory and sends them together in a single request.

Now the flow becomes: 

Producer Buffer

msg1
msg2
msg3
msg4
msg5
   ↓
BATCH
   ↓
single network request


Producer
   ↓
Batch [msg1 msg2 msg3 msg4 msg5]
   ↓
Kafka Broker

Instead of 5 requests, we send 1 request. Much faster.

>> Where Does Batching Happen?

Batching happens inside the producer application, in memory. The producer maintains buffers for each partition.

Example: 

Producer Memory

Partition 0 buffer
msg1
msg3
msg7

Partition 1 buffer
msg2
msg4
msg6

Each partition gets its own batch. When a batch is ready, Kafka sends it to the leader broker of that partition.

>> Two Configurations Control Batching

Two producer settings determine when batches are sent. batch.size Maximum size of a batch in bytes. linger.ms How long the producer waits for more messages before sending the batch.

If batch.size is 1MB and linger.ms is 100ms, the producer will send a batch when either:
  • the batch reaches the maximum size (batch.size)
  • the linger time expires (linger.ms)

Consumer Polling and Fetching

Consumers fetch batches of messages. Important settings:
  • max.poll.records: This controls how many records returned in a single poll() call.

  • fetch.min.bytes: This controls the minimum amount of data that must be available before a responding to a fetch request. This means if fetch.min.bytes: 5000 and a consumer sends a fetch request, the broker will wait until at least 5000 bytes of data are available.

  • fetch.max.wait.ms: This setting defines the maximum time the broker will wait before returning whatever data is available. So if fetch.min.bytes is not met within the fetch.max.wait.ms time, the broker will return whatever data is available, even if it is less than fetch.min.bytes.

  • fetch.max.bytes: This controls the maximum amount of data that can be fetched in a single request by a consumer.

Although you might be thinking that if we already have max.poll.records why do we need fetch.max.bytes?

The answer is something called consumer fetch buffer. When a consumer fetches data, kafka returns fetch.max.bytes size of data which is stored in consumer fetch buffer. When a conumer poll() is done, from this buffer, max.poll.records number of records are returned to the consumer application.

The flow looks something like this: 
Broker
  │
  │ (fetch response)
  ▼
Consumer Fetch Buffer
(max size controlled by fetch.max.bytes)
  │
  │ poll()
  ▼
Application
(max records controlled by max.poll.records)

Heartbeats and Rebalancing

Consumers also send heartbeats to Kafka to show they are alive.

If a consumer stops heartbeating, Kafka triggers rebalance.

Rebalacing means redistributing the partitions among the available consumers.

Why Kafka Is So Fast

Kafka performance comes from several design choices:
  • append-only log storage
  • sequential disk writes
  • zero-copy network transfer
  • partition parallelism
  • batching and compression

These allow Kafka clusters to handle millions of events per second.

Final Thoughts

Kafka started as a messaging system. Today it powers:
  • event-driven architectures
  • real-time analytics
  • stream processing pipelines
  • microservice communication

At its core Kafka solves one simple problem: moving large amounts of data safely between systems.

Once you understand topics, partitions, producers, and consumer groups, the rest of Kafka becomes surprisingly logical. And that is the moment when distributed systems stop looking terrifying and start looking fascinating. Or at least fascinating enough to keep reading.

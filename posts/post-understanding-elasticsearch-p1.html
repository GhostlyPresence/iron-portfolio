Understanding Elasticsearch — Part 1 (For Absolute Beginners)

A gentle introduction to how Elasticsearch works and why modern applications rely on it for fast search.

If you've ever searched for a product on Amazon, looked for a log in an observability dashboard, or filtered results on an e-commerce site, you were likely interacting with a search engine such as Elasticsearch.

Elasticsearch is one of the most widely used search technologies in modern software systems. Yet for many engineers encountering it for the first time, it can feel confusing. Is it a database? A search engine? A distributed system?

The answer is: a little bit of all three.

What Elasticsearch Really Is

Elasticsearch is a distributed search and analytics engine. It is built to store large amounts of data and allow users to search that data very fast.

Instead of scanning every document when you search, Elasticsearch builds special data structures that allow it to jump directly to the relevant results.

This is why queries that would normally take seconds or minutes in a traditional database can often be answered in milliseconds.

Documents and Indices

The basic unit of data in Elasticsearch is a document. A document is simply a JSON object that represents a piece of data. 


        {
          "title": "Running Shoes",
          "brand": "Nike",
          "price": 1999,
          "in_stock": true
        }

Documents are stored inside something called an index. You can think of an index as a collection of related documents, similar to a table in a traditional database.

Why Elasticsearch Is Fast

The reason Elasticsearch is fast comes down to something called an inverted index.

Instead of storing documents in a way that requires scanning each one, Elasticsearch builds a structure that maps words to the documents that contain them.

For example:


          running → [doc1, doc2]
          shoes   → [doc1, doc2, doc3]
          leather → [doc3]

When someone searches for “running shoes”, Elasticsearch simply looks up both words and finds the documents that contain them.

This approach makes search extremely fast even with millions of documents.

Keyword vs Text Fields

When storing data in Elasticsearch, fields are usually mapped as either keyword or text.

A keyword field stores the value exactly as it is. It is useful for filtering, sorting, and aggregations.
A text field, on the other hand, is analyzed and broken into searchable tokens. This makes it useful for full-text search.

Say we have the follwing document: 


{
  "city": "New Delhi",                  ---- this field is mapped as keyword data type
  "description": "Capital of India"     ---- this field is mapped as text data type
}

The tokens generated will look like this: 


"New Delhi" --> [doc1]          --> The two words will be indexed together as a single token "New Delhi"
"Capital"   --> [doc1]          --> The word "Capital" will be indexed as a token
"of"        --> [doc1]          --> The word "of" will be indexed as a token
"India"     --> [doc1]          --> The word "India" will be indexed as a token
While searching, if we search using a field that is a keyword, it has to be an exact match. So if we search for "New Delhi" in the city field, it will match [doc1].
But if we search for "Delhi" in the city field, it will not match [doc1] because the city field is indexed as a keyword and requires an exact match.

But for the description field, if we search for "Capital", it will match [doc1] because the description field is indexed as text and is analyzed into tokens.

Other Data Types

In addition to keyword and text, Elasticsearch supports many other data types such as numbers, dates, booleans, geo points, and more.

Elasticsearch does not create a token-based inverted index for numeric or date fields. Instead it stores them using special numeric indexing structures (BKD trees) optimized for range queries.

Why?
Because numeric queries are usually range queries.

Typical queries: 
price > 50
price between 50 and 200
If Elasticsearch used a normal inverted index, it would need entries like: 

100 → doc1
101 → doc2
102 → doc3
That works for exact matches, but it is terrible for ranges.

Example query: 


price between 50 and 200
It would need to scan 150 index entries. So Elasticsearch uses BKD (Balanced K-Dimensional Tree) trees instead.

Distributed by Design

Elasticsearch is also designed to run across multiple machines. Data is split into pieces called shards, which are distributed across nodes in a cluster.

This allows Elasticsearch to scale horizontally. Instead of one server doing all the work, many servers can process searches in parallel.

To ensure reliability, Elasticsearch also keeps replica shards. If one node fails, another node can immediately take over.

Why So Many Systems Use Elasticsearch

Elasticsearch has become a standard part of many modern architectures because it solves a very common problem:

Applications need to search large datasets quickly while supporting filtering, ranking, and analytics.

Instead of building a search system from scratch, teams rely on Elasticsearch to handle that complexity.
